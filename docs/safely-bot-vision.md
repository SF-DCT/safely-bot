作成日: 2026年4月1日
作成者: 高橋幹佳（BGS事業部 General Manager）

## 1. プロジェクト概要

### 1.1 背景と目的

株式会社SAFELYのBGS事業部では、事業成長に伴い業務の複雑性が増大している。General Managerとして、広告運用・SEO/AIO分析・チーム管理・クライアント対応・社内レポーティングなど多岐にわたる業務を効率化するため、Slack上で動作するAI秘書ボット「safely-bot」を構築する。

### 1.2 コンセプト

「Slackから離れずに、あらゆる業務を完結させる」

safely-botは単なる日報自動化ツールではなく、情報検索・ドラフト作成・タスク実行までをカバーするAI秘書プラットフォームとして段階的に進化させる。すべての実行アクションには「確認→承認」フローを必須とし、人間が最終判断を下す運用を徹底する。

### 1.3 技術基盤

- Runtime: Node.js + TypeScript
- Slack連携: Slack Bolt for JavaScript（Socket Mode）
- AI: Anthropic API（Claude Sonnet 4） — tool use（Function Calling）によるインテント分類とツール選択
- スケジューラー: node-cron（平日定時実行）
- ホスティング: Railway / Render / Fly.io（月額約$5〜）
- 構造: モジュラー設計 — 機能ごとにツールモジュールを追加していくプラグイン方式

## 2. アーキテクチャ

### 2.1 コアエンジン（全フェーズ共通）

safely-botの心臓部は以下の3つのコンポーネントで構成される。

**Intent Router（意図分類器）**
Slack DMで受け取ったメッセージをClaudeが解析し、どのツールを呼び出すべきかを判断する。「日報書いて」「ISCLの広告費教えて」「明日のMTG入れて」など、自然言語での指示をtool useで適切なモジュールにルーティングする。

**Tool Dispatcher（ツール実行器）**
Intent Routerが選択したツールを実行し、各APIへのリクエストを処理する。ツールはモジュールとして独立しており、新機能追加時は新しいツールモジュールを登録するだけで拡張可能。

**Approval Flow（承認フロー）**
すべての「書き込み」操作（メール送信、カレンダー登録、広告変更、投稿など）に対して、Slack Block Kit UIで確認画面を表示する。「承認して実行」「修正して再生成」「キャンセル」の3択を提示し、人間の承認なしに外部システムへの変更は行わない。

### 2.2 データフロー

```
Slack DM / 定時トリガー
    ↓
Intent Router（Claude tool use）
    ↓
Tool Dispatcher → 各API呼び出し
    ↓
結果をClaudeが整形・分析
    ↓
Slack DMで確認依頼（Block Kit UI）
    ↓
[承認] → 実行 + ログ記録
[修正] → 再生成ループ
[キャンセル] → 終了
```

## 3. フェーズ別ロードマップ

### Phase 2: 日報自動化（構築中）+ インテリジェンスブリーフィング（✅ 稼働中）

**目標:** 毎日19:00 JSTに日報ドラフトを自動生成し、確認後に#日報チャンネルに投稿する。

**現在の到達点:**
- Phase 1（Claude Codeスキル）: 完成済み。`/daily-report`コマンドで動作するが、Claude Codeセッション中のみ。
- Phase 2 Step 1: 完了。Slack BotがSocket Modeで接続し、DMに応答できる状態。
- **インテリジェンスブリーフィング: ✅ 実装完了・Railway稼働中。** 毎朝9:00 JSTに6トピック（SEO/広告/Webマーケ/マネジメント/AI/営業）の最新情報をYouTube API+RSSで収集、Claude APIでキュレーション・要約してSlack DMに自動配信。DMで「ブリーフィング」と送ると手動実行も可能。

**残りのステップ:**

| Step | 内容 | 難易度 | 見積もり |
|------|------|--------|----------|
| Step 2 | データ収集モジュール（6データソース） | 中 | 3-5日 |
| Step 3 | Claude APIで日報ドラフト生成 | 中 | 2-3日 |
| Step 4 | Slack DM確認フロー（Block Kit UI） | 中 | 2-3日 |
| Step 5 | node-cronスケジューラー + デプロイ | 低 | 1-2日 |

**データソース:**
- Slack活動ログ: 今日の自分のメッセージを`search.messages`で取得
- 前日日報: #日報-高橋幹佳から「今後行う業務」を抽出 → 今日のTRY欄に反映
- 自分宛DMメモ: 業務中に気づいたことをメモ → 自動反映
- 岡野社長の発信: lab-xxチャンネル → 「上司からのFB」欄に活用
- Googleカレンダー: 今日のMTG + 明日の予定
- Notion: タスク・ノート

**日報フォーマット（SAFELY定型）:**
1. 前日設定したTRY
2. 本日行った業務（MTG / IS / SF / TC / Another / Routine）
3. 本日の業務での不足／不足を埋めるための行動
4. 上司からのFBで得た気付き／ネクストアクション
5. 気付いたこと／良かったこと／継続していきたいこと
6. 今後行う業務（ルーティン＋明日のカレンダー）
7. その他

### Phase 3: AI秘書化（次期開発）

Phase 2の日報機能をベースに、safely-botを汎用AI秘書へ拡張する。DMでの自由な質問・指示に対応し、複数のツールを使い分ける。

#### 3.1 情報検索系（Read — すぐ実装可能）

**Slack知識検索**
- 難易度: 低（2-3日）
- API: Slack `search.messages`（既存のBot Tokenで対応可能）
- ユースケース: 「先週の金脈会議の決定事項教えて」「愛嶋さんと先月話したクリエイティブの件まとめて」
- 実装: 意図検出 → キーワード抽出 → Slack検索 → Claudeで要約

**Notion知識検索**
- 難易度: 低（2-3日）
- API: Notion API search + database query
- ユースケース: 「先月のBGS定例の議事録まとめて」「水道修理PFの改善タスク一覧見せて」
- 備考: Phase 2でNotion APIトークンを設定済みの場合、追加認証不要

#### 3.2 Google Ads運用（Read + Write — 最重要拡張）

広告運用の自動化は、高橋さんの業務時間に最もインパクトがある拡張機能。

**広告パフォーマンスレポート**
- 難易度: 中（1週間、トークン承認後）
- API: google-ads-api（npm v23.0.0）、GAQL
- ユースケース:
  - 毎朝の全クライアントサマリー自動配信
  - 「ISCLの先週のCV数とCPAの推移は？」へのオンデマンド回答
  - P-MAXアセットグループ別パフォーマンス取得
- 前提条件: Google Ads APIデベロッパートークンの取得（MCC > ツールと設定 > APIセンターから申請。審査に数日〜1週間）

**問題自動診断**
- 難易度: 中（1週間）
- ロジック: Claudeがメトリクスを分析し、閾値・トレンドと比較
- 検知パターン:
  - 予算消化率の異常低下（target ROAS過達による配信抑制）
  - CPA急騰（検索語句汚染、品質スコア低下）
  - インプレッションシェア低下（競合入札強化、予算不足）
- 出力: 問題の特定 + 根本原因分析 + 改善案2-3件の提示

**広告アカウント操作**
- 難易度: 高（2週間）
- API: MutateOperation（campaign_budget, campaign, ad_group_criterion）
- 操作例:
  - キャンペーン予算の変更
  - 入札戦略（target ROAS / target CPA）の調整
  - キーワードの追加・除外
  - キャンペーンの一時停止・再開
- 安全対策: 全変更にDM承認必須。#ads-operationsチャンネルに変更ログを自動投稿。予算・入札変更にはロールバック機能を実装。テストアカウントでの十分な検証を経てから本番適用。
- MCC対応: デベロッパートークン1つで複数クライアントアカウント（ISCL-クリーンライフ、ハチ駆除等）を横断管理可能

#### 3.3 コミュニケーション系（Draft + Execute）

**Gmail返信ドラフト**
- 難易度: 中（1週間）
- API: Gmail API（messages.list, messages.get, drafts.create, messages.send）
- ユースケース: 「今日の未読メール3通の返信ドラフト作って」「このメールに丁寧にお断りの返信書いて」
- 安全対策: 必ずドラフトとして作成し、送信前に宛先・件名・本文をDMで確認

**カレンダーイベント作成**
- 難易度: 低（2-3日）
- API: Google Calendar API `events.insert`
- ユースケース: 「明日14時から田尾さんと30分MTG入れて」「来週水曜に事業成長戦略チームの定例を設定して」
- 備考: gws-cliで既にカレンダー読み取りは成功済み。書き込みスコープの追加のみで対応可能

#### 3.4 タスク管理系

**Asanaタスク操作**
- 難易度: 低（2-3日）
- API: Asana API（claude.aiでMCP接続済み）
- ユースケース: 「水道修理の売上回復分析のタスク作って、期限は来週金曜」「今週の未完了タスク教えて」

#### 3.5 外部システム連携

**Salesforceデータアクセス**
- 難易度: 高（2週間）
- API: Salesforce REST API（SOQL, sObject）
- ユースケース: 「ISCLの今月のSF数値まとめて」「新規リードを登録して」
- 備考: OAuth 2.0 Connected Appの設定が複雑。まずはRead（SOQL検索）のみ実装し、Write（レコード作成）は後から追加

**WordPress記事管理**
- 難易度: 中（1週間）
- API: WP REST API（/wp-json/wp/v2/posts, /pages）
- ユースケース: 「水道修理セーフリーのFAQページにこの内容追加して」「粗大ゴミ回収本舗のブログ記事を下書きで作って」
- 安全対策: 常に下書き（draft）として作成し、公開前にDMで確認。クライアントサイトへの変更は影響が大きいため、特に慎重な承認フローを適用
- 認証: クライアントサイトごとにApplication Passwordsを設定

## 5. セキュリティと運用ポリシー

### 5.1 認証・権限管理

- すべてのAPIトークン・シークレットは環境変数（`.env`）で管理し、コードにハードコードしない
- Google系サービスのOAuth Refresh Tokenは暗号化して保存
- Slackのsigning secretによるリクエスト検証を必ず実施
- クライアントサイト（WordPress等）へのアクセスは最小権限の原則を適用

### 5.2 実行安全性

- 外部システムへの「書き込み」操作はすべてDM承認を必須とする
- Google Ads変更は#ads-operationsチャンネルに変更ログを自動記録
- 予算・入札の変更にはロールバック機能を実装（変更前の値を保持）
- クライアントサイトへの投稿は常にdraftステータスで作成

### 5.3 監視・ログ

- すべてのAPI呼び出しと実行結果をログに記録
- エラー発生時はSlack DMで即時通知
- 月次でAPI利用量とコストをレビュー

## 7. 技術的な注意事項

### Google Ads APIデベロッパートークン申請

- MCC管理画面 > ツールと設定 > APIセンターから申請
- 申請時にAPIの利用目的を記載（自社広告運用の自動化・レポーティング）
- 審査には数日〜1週間程度かかる場合がある
- Phase 2実装と並行して早めに申請を開始すること

### Intent Router設計のポイント

ClaudeのFunction Calling（tool use）を使い、DMメッセージに対してどのツールを呼び出すかを判断する。ツール定義の例:

- `generate_daily_report`: 日報ドラフトを生成する
- `search_slack`: Slackのメッセージを検索して要約する
- `get_ads_performance`: Google Adsのパフォーマンスデータを取得する
- `diagnose_ads_issue`: 広告の問題を診断して改善案を提示する
- `apply_ads_change`: 承認された広告変更を実行する
- `create_calendar_event`: Googleカレンダーにイベントを登録する
- `draft_email_reply`: Gmailの返信ドラフトを作成する
- `create_asana_task`: Asanaにタスクを作成する
- `search_notion`: Notionを検索して情報を取得する
- `publish_wp_draft`: WordPressに下書き記事を作成する

ツール追加時はこのリストに新しい定義を追加するだけで、コアロジックの変更は不要。
